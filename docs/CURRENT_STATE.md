**What changed in code (revert tag `pre-leap-0.10.9-migration`):**
- **SPM** (`project.pbxproj` + `Package.resolved`): `leap-ios`@`0.9.4` → `leap-sdk.git`@`0.10.9`; product `LeapSDK` → `LeapModelDownloader`. Needed for device deploy; a no-op for sim build/test.
- **`LFM2ModelLoader.swift`** — bundled-GGUF load via `Leap.shared.load(url:options:generationTimeParameters:autoDetectCompanionFiles:)` (D2); `LiquidCacheOptions.enabled(path:)` (D3, retires `cacheMaxEntries`); sampling params stay **per-call** via `GenerationOptions().with(temperature:)…`; `onEnum(of:)` response mapping. **New in v0.10.9:** the generation stream can terminate with `MessageResponseError` (`.error`); the adapter surfaces it as a thrown stream finish (`TranslationError.generationFailed`), the one behavioural change vs the p2 reference (which drops `.error`).

The destructive button's deferred `Task` read `model.pending` inside `confirmPending()`, but SwiftUI dismisses the dialog by flipping the `isPresented` binding, whose setter synchronously runs `cancelPending()` (`pending = nil`) **before** the deferred Task runs — so `confirmPending()` hit `guard let action = pending` with `nil` and returned without acting. Fix (`SettingsView.swift` + `SettingsViewModel.swift`): capture the pending action **synchronously at tap time** and pass it into `confirmPending(_:)`; re-entry safety now rests on the busy-flag guard (`isClearingCache` / `isDeletingAll`) as the method's first line + `@MainActor` serialization. Doc-comments updated to document the race and the new re-entry mechanism.

## Production wiring gap discovered 2026-05-25 — ✅ RESOLVED (B1.6, `32abc3e`)

P0 finding from the 2026-05-25 reconciliation pass, **fixed the same day (B1.6, commit `32abc3e`)**. Was tracked as **B1.6** in `docs/PRE_MVP1_REVIEW.md` Bucket 1.

- **The bug:** `ArigatoAIApp.swift:41` had constructed the app's `ModelContainer` with `Schema([Item.self])` — the default Xcode scaffold entity. `Meeting` and `Sentence` were never registered in the production schema.
- **Why it reached the store:** `MeetingStore` (`@ModelActor`) is constructed against that exact container at `AppBootstrapper.swift:598` (`MeetingStore(modelContainer: container)`). With `Meeting`/`Sentence` absent from the model, the first production `startMeeting` → `insert(Meeting)` → `save()` would have failed at runtime.
- **Only appearance of the entities in production (pre-fix):** a `#if DEBUG` preview at `MeetingControlsView.swift:791` (`ModelContainer(for: Meeting.self, Sentence.self, …)`) — never the shipping path.
- **Why it never surfaced:** the gap was masked by the B1.1 LFM2 block. Store construction is gated on `lfm2Loader.warmup()` succeeding (`AppBootstrapper.swift:560` → `:598`); while LFM2 was down, `meetingStore`/`coordinator` stayed `nil`, the controls fell back to `.disabled()`, and the Settings gear was hidden — so no persistence path was reachable on device. The B1.5 debug bypass renders the UI shell but does not construct the store.
- **Why tests stayed green:** every persistence test injects its own in-memory `ModelContainer` built with the correct `Meeting`/`Sentence` schema, so the production-container construction path was exercised by no test until B1.6 added `ArigatoAIAppTests.productionContainer_registersMeetingAndSentence`. Process gap V3-filed ("Schema-registration test gap").
- **Fix (B1.6, shipped `32abc3e` 2026-05-25, independent of B1.1):** replaced `Schema([Item.self])` with `Schema([Meeting.self, Sentence.self])` behind a `makeAppSchema()` static factory, deleted the `Item.swift` scaffold, added `ArigatoAIAppTests.productionContainer_registersMeetingAndSentence` asserting `Meeting` + `Sentence` registration (and exactly two entities). Build clean; serial suite 399/0/2.
