> **⛔ DO NOT verify leap-sdk APIs against the `.swiftinterface` / `.abi.json` — COMPILE instead. ⛔**
> leap-sdk is a **SKIE / Kotlin-Native bridged** SDK. Its idiomatic Swift convenience overloads — `ModelRunner.createConversation(systemPrompt:)`, the `generateResponse(...)` overloads, the `GenerationOptions().with(...)` builders, the `onEnum(of:)` sealed-enum matcher, and the `MessageResponse` case payloads — are **generated by SKIE OUTSIDE the `.swiftinterface` file.** Grepping the interface for these symbols gives **FALSE NEGATIVES.** During this migration that false-negative trap cost ~5 wasted verification dispatches ("the API doesn't exist" — it did). **The compiler is the only source of truth for this SDK's Swift surface.** If you need to confirm a call shape exists, write it and build; do not grep the interface. The device-proven reference build `~/AI-projects/arigato-ai-p2` (leap-sdk 0.10.9, rev `a83ca1e`) compiles clean and deploys to hardware — every call shape it uses is proven to exist in 0.10.9 by that compilation.

# LEAP iOS SDK — LFM2-350M-ENJP-MT integration (v0.10.9)

## Critical system prompt requirement
LFM2-350M-ENJP-MT requires ONE of these EXACT system prompts. No variation, no rephrasing:
- `"Translate to English."` — for Japanese → English
- `"Translate to Japanese."` — for English → Japanese

The user turn is the text to translate. Single-turn conversations only — do not accumulate history across translation calls.

## SDK package setup (v0.10.9)

**Package URL:** `https://github.com/Liquid4All/leap-sdk.git`

**Pinned product:** `LeapModelDownloader` — this is the only product we link (it transitively brings `LeapSDK`'s symbols: `Leap`, `ModelRunner`, `Conversation`, `GenerationOptions`, `MessageResponse`, `onEnum(of:)`, etc.). Do NOT also link a separate `LeapSDK` product — the v0.10.6+ dual-import guard makes linking both an error.

**Swift import:** `import LeapModelDownloader`

**Nested-dylib re-sign (DEVICE BUILDS ONLY).** `LeapModelDownloader.framework` embeds nested dynamic libraries under its own `Frameworks/` directory. On-device codesigning requires those nested dylibs (and the framework) to be re-signed with the app's expanded signing identity, or the device install fails. The app target carries a final **Run Script build phase** that does this, **platform-gated to `iphoneos` so it is INERT on the simulator** (sim build/test never runs it). The script:

```bash
if [ "${PLATFORM_NAME}" != "iphoneos" ]; then exit 0; fi
FW="${CODESIGNING_FOLDER_PATH}/Frameworks/LeapModelDownloader.framework/Frameworks"
shopt -s nullglob; d=("$FW"/*.dylib); [ ${#d[@]} -eq 0 ] && { echo "error: no nested dylibs"; exit 1; }
for f in "${d[@]}"; do codesign --force --sign "${EXPANDED_CODE_SIGN_IDENTITY}" --timestamp=none "$f" || exit 1; done
codesign --force --sign "${EXPANDED_CODE_SIGN_IDENTITY}" "${CODESIGNING_FOLDER_PATH}/Frameworks/LeapModelDownloader.framework"
```

If a device install starts failing with a codesign/`@rpath` error on a nested dylib, check this phase first.

## Model loading pattern (v0.10.9) — bundled GGUF

This project ships the GGUF **bundled in the app** (`LFM2-350M-ENJP-MT-Q5_K_M.gguf` in `Bundle.main`) and loads it by URL — there is **no portal download** and no download-progress channel.

```swift
import LeapModelDownloader

guard let ggufURL = Bundle.main.url(
    forResource: "LFM2-350M-ENJP-MT-Q5_K_M",
    withExtension: "gguf"
) else { /* throw modelLoadFailed */ }

let options = LiquidInferenceEngineOptions(
    bundlePath: ggufURL.path,
    cacheOptions: LiquidCacheOptions.enabled(path: cachePath),
    cpuThreads: nil, contextSize: nil, nGpuLayers: nil,
    mmProjPath: nil, audioDecoderPath: nil, chatTemplate: nil,
    audioTokenizerPath: nil, audioDecoderUseGpu: false,
    useMmap: nil, extras: nil
)

let runner: any ModelRunner = try await Leap.shared.load(
    url: ggufURL,
    options: options,
    generationTimeParameters: nil,
    autoDetectCompanionFiles: false
)
```

- `Leap.shared.load(url:...)` (instance method on the `Leap.shared` singleton) is the bundled-file entrypoint. The v0.9.4 `Leap.load(model:quantization:...)` portal entrypoint is gone.
- `autoDetectCompanionFiles: false` — the GGUF is self-contained (no sidecar projector/tokenizer).
- `generationTimeParameters: nil` — **sampling params stay per-call** (see streaming section), not load-time.

## KV cache config (v0.10.9)

```swift
let cacheOptions = LiquidCacheOptions.enabled(path: cachePath)
```

`LiquidCacheOptions` is now a factory-style type exposing **only** `.enabled(path:)` — the v0.9.4 `LiquidCacheOptions(path:maxEntries:)` struct init (and the `maxEntries` cap) is **gone**. Pass the result via `LiquidInferenceEngineOptions(... cacheOptions: ...)`. The cache path is the user Caches directory (`LFM2CachePathResolver.resolve()`) — NOT iCloud-backed; OS-purgeable; rebuilt on next translation. On-device benchmark is V3-tracked under "LFM2 prompt cache effectiveness benchmark."

## Conversation creation and streaming (v0.10.9)

```swift
let conversation = runner.createConversation(systemPrompt: "Translate to English.")

// Per-call sampling: zero-arg init + chained .with(...) builders.
// The v0.9.4 value-init GenerationOptions(temperature:topP:...) is GONE.
let options = GenerationOptions()
    .with(temperature: Float(0.3))
    .with(topP: Float(1.0))
    .with(minP: Float(0.15))
    .with(repetitionPenalty: Float(1.05))

let stream = conversation.generateResponse(
    userTextMessage: inputText,
    generationOptions: options
)
```

For a warmup canary with no options, use the message overload:

```swift
let message = ChatMessage(role: .user, content: .text("Hello."))
let stream = conversation.generateResponse(message: message, generationOptions: nil)
```

> **`ChatMessage` content:** v0.10.9 takes a **single** `.text(_:)` content (`content: .text(...)`), not a v0.9.4-style array (`content: [.text(...)]`).

**Maintainer-recommended sampling defaults** (from the published manifest): `temperature: 0.3`, `min_p: 0.15`, `repetition_penalty: 1.05`. These live in `TranslationGenerationParameters.recommended`.

**`MessageResponse` case handling in v0.10.9 — use `onEnum(of:)`:**

```swift
for try await response in stream {
    switch onEnum(of: response) {
    case let .chunk(chunk):
        accumulated += chunk.text          // payload is MessageResponseChunk, NOT a bare String
    case .complete:
        // terminal success
    case let .error(e):
        // NEW in v0.10.9: MessageResponseError. Surface it as a failure —
        // do NOT silently drop it (a dropped .error looks like a truncated
        // translation). This project finishes the stream by throwing
        // TranslationError.generationFailed(String(describing: e)).
    case .reasoningChunk, .functionCalls, .audioSample:
        continue                           // dropped per seam scope
    }
}
```

Key deltas from v0.9.4:
- `MessageResponse` is a **SKIE sealed type** — case-match goes through the `onEnum(of:)` free function (a plain `switch response` over bare cases no longer compiles). **Do not grep the interface to confirm the cases — compile.**
- `.chunk`'s payload is a **`MessageResponseChunk`** with `.text: String` (v0.9.4 was a bare `String`).
- `.functionCalls` is **plural** in v0.10.9.
- `.error` (`MessageResponseError`) is **new** — it is constructible in tests as `MessageResponseError(throwable: KotlinThrowable(message:))`.

## Direction handling

```swift
enum TranslationDirection {
    case jaToEn, enToJa

    var systemPrompt: String {
        switch self {
        case .jaToEn: return "Translate to English."
        case .enToJa: return "Translate to Japanese."
        }
    }
}
```

## Cold-start mitigation

No SDK-provided `prewarm()` primitive. Use a dummy inference per direction at app launch (the project's `LFM2ModelLoader.warmup()` runs both canaries sequentially: EN→JA `"Hello."`, then JA→EN `"こんにちは。"`).

## Concurrency constraints (v0.10.9)

- `ModelRunner` is a SKIE/Kotlin-Native-bridged protocol with **no** `Sendable` inheritance; `Conversation` is a class with no `Sendable` conformance — **unchanged from v0.9.4**.
- The established `@unchecked Sendable` adapter pattern (`LFM2EngineAdapter`) is required for crossing actor boundaries, with one actor (`LFM2ModelLoader`) the sole owner serialising every call.
- Construct a fresh `Conversation` per call via `runner.createConversation(systemPrompt:)` so there is no cross-call mutable SDK state.

**Scheduling assumption + violation behaviour.** The adapter assumes its owning actor drives at most one inference at a time. If that invariant is violated (two `translate(...)` streams against the same adapter concurrently), the two SDK generations interleave on the SDK's internal executor with **undefined token ordering** — no data race (each `Conversation` is independent), but output coherence is not guaranteed. Serialisation by the owning actor is what keeps output coherent.

**Cancellation (v0.10.9):** Cooperative. Cancelling the Swift Task breaks the `for try await` loop and drops the SDK generation reference; the SDK reports the abandoned generation as **`.interrupted`** and the adapter finishes the stream **without** a trailing `.complete` (no partial result is fabricated). `GenerationFinishReason` has no `.cancelled` case; cancellation surfaces as a clean stream finish.

## What this model is good at (from model card)
- Short-to-medium text (optimized for low latency at sentence scale)
- Bidirectional Japanese ↔ English
- Preserves tone in natural speech

## What it's NOT good at (from model card)
- Long-form (>3 paragraphs) — chunk first
- Single-turn only — do not feed it multi-turn history

## History — the v0.9.4 → v0.10.9 migration

The project shipped on `Liquid4All/leap-ios` v0.9.4 through MVP-1 while the v0.10.x `leap-sdk` channel was upstream-blocked on an @rpath XCFramework packaging bug ([Liquid4All/leap-sdk#5](https://github.com/Liquid4All/leap-sdk/issues/5)), then briefly "not adopting" after the fix shipped in v0.10.9 but before it was binary/device-verified here. **Migrated 2026-06-06** once the reference build `~/AI-projects/arigato-ai-p2` proved v0.10.9 compiles clean and deploys to a physical iPhone. The breaking Swift deltas absorbed in the migration: `LeapSDK` → `LeapModelDownloader` product/import; `Leap.load(model:)` → `Leap.shared.load(url:)`; `GenerationOptions(value-init)` → `GenerationOptions().with(...)`; `LiquidCacheOptions(path:maxEntries:)` → `.enabled(path:)`; bare-`switch` `MessageResponse` → `onEnum(of:)` + `MessageResponseChunk.text`; new `.error` terminal; `ChatMessage` content array → single `.text(_:)`. The portal-download / `LeapDownloaderConfig` surface is **not exercised** (bundled GGUF). Call-site inventory + P-2 execution history: `docs/PHASE_5_B1_1_MIGRATION_INVENTORY.md`.
